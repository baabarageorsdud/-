## 第1页：项目概述

### 1.1 选题背景与意义

在数字化时代，学历学位证书是个人职业发展的重要依据。传统认证方式存在以下痛点：

- **效率低、流程繁琐**：通过学信网等平台需经过多轮人工审核，周期长。
- **中心化风险**：一旦中心数据库遭受攻击或内部管理不善，可能导致数据泄露、篡改。
- **溯源能力不足**：难以追溯证书的完整颁发链路，造假成本低。
- **移动端体验差**：现有系统以网页端为主，移动端适配不足。

本项目以**微信小程序**为入口，利用**以太坊区块链**进行链上存证，构建一套轻量级、可追溯的学历学位认证系统，旨在解决上述痛点。

### 1.2 项目目标

| 目标 | 描述 |
|------|------|
| 便捷性 | 用户无需安装额外 App，通过微信即可提交认证申请、查询进度 |
| 可信度 | 关键认证数据的 hash 写入以太坊区块链，不可篡改、可追溯 |
| 可验真 | 任何人持有原始字段均可本地重算 hash 与链上对比完成验真 |

### 1.3 个人工作说明

本项目由本人独立完成，涵盖：需求分析、数据库设计、后端接口设计与实现、微信小程序端页面开发、区块链存证方案设计与实现（Solidity 合约 + Spring Boot web3j 集成）、测试与文档撰写。

## 第2页：系统架构与技术栈

### 2.1 总体架构（前后端分离 + 区块链存证）

```
┌──────────────────────────────────────────────────────┐
│                 微信小程序（前端）                      │
│  WXML / WXSS / JS   wx.request → REST API            │
└──────────────────────┬───────────────────────────────┘
                       │ HTTP
┌──────────────────────▼───────────────────────────────┐
│              Spring Boot 后端（业务层）                 │
│  Controller → Service → MyBatis Mapper → MySQL        │
│            ↓ web3j（异步）                             │
│  AttestationService → 以太坊节点 RPC                  │
└──────────────────────┬───────────────────────────────┘
                       │ JSON-RPC
┌──────────────────────▼───────────────────────────────┐
│              以太坊节点（本地 Hardhat / 测试网）         │
│  Attestation.sol 合约                                 │
│  attest(requestId, bytes32 hash, meta)                │
│  getAttestation(requestId) → (hash, ts, issuer, meta) │
└──────────────────────────────────────────────────────┘
```

### 2.2 技术栈

| 层次 | 技术选型 | 说明 |
|------|---------|------|
| 前端 | 微信小程序（原生） | WXML/WXSS/JS，调用 wx.request |
| 后端框架 | Spring Boot 3.x | REST API，Controller-Service-Mapper 分层 |
| ORM | MyBatis | XML Mapper，灵活的 SQL 控制 |
| 数据库 | MySQL | 业务数据存储 |
| 区块链 | 以太坊 + Solidity 0.8 | 链上存证，Hardhat 本地开发环境 |
| 区块链 SDK | web3j 4.x | Java 调用以太坊节点 |
| 合约开发 | Hardhat | 编译、部署、测试 |

### 2.3 核心业务流程

```
用户提交申请
    │
    ▼
后端入库（MySQL）
    │
    ▼
计算 canonical hash（SHA-256）
    │
    ├─→ 写入 MySQL blockchain_hash 字段
    │
    └─→ 调用 Attestation.attest()  ──→  链上存证（不可篡改）
                                         └─→ 触发 Attested 事件
    ▼
返回 txHash + attestedHash 给前端
    │
    ▼
用户/第三方验真：GET /api/attest/{requestId}
    └─→ 从链上读取 hash，与本地重算对比
```

### 4.2 区块链存证实现流程

**写入存证（attest）：**

```
1. 后端接收申请，生成 canonical string：
   "requestId|userId|schoolId|degree|major|graduationYear|timestamp"

2. SHA-256(canonical) → bytes32 hash

3. 写入 MySQL（blockchain_hash = hash, blockchain_status = PENDING）

4. web3j 调用 Attestation.attest(requestId, hash, meta)
   └→ 合约验证 onlyOwner，写入 mapping，触发 Attested 事件

5. 交易确认后，更新 MySQL blockchain_tx = txHash, blockchain_status = SUCCESS
```

**验证存证（verify）：**

```
1. 验证方提供 requestId 和原始字段
2. 本地重算 canonical hash
3. GET /api/attest/{requestId} 从链上读取 storedHash
4. 比对：localHash == storedHash → 验真通过
```

### 5.1 微信登录实现

```javascript
// 小程序端
wx.login({
  success(res) {
    wx.request({
      url: '/api/user/wechat-login',
      method: 'POST',
      data: { code: res.code },
      success(loginRes) {
        // 存储 token 到 storage
        wx.setStorageSync('token', loginRes.data.token);
        wx.setStorageSync('userInfo', loginRes.data.userInfo);
      }
    });
  }
});
```

```java
// Spring Boot 后端
@PostMapping("/wechat-login")
public ResponseEntity<?> wechatLogin(@RequestBody Map<String, String> body) {
    String code = body.get("code");
    // 调用微信接口换取 openid
    String openid = wechatService.getOpenid(code);
    // 查询或创建用户
    User user = userService.findOrCreateByOpenid(openid);
    // 生成 JWT token
    String token = jwtService.generateToken(user);
    return ResponseEntity.ok(Map.of("token", token, "userInfo", user));
}
```

## 第6页：创新点、不足、展望与面试提纲

### 6.1 创新点与亮点

1. **区块链存证落地实现**：本仓库提供完整的本地以太坊（Hardhat）+ Solidity 合约 + Spring Boot（web3j）的最小存证实现，而非仅停留在设计阶段。
2. **隐私优先的 hash 存证设计**：链上只存 SHA-256 摘要，既保护用户隐私，又实现防篡改可追溯。
3. **微信小程序入口**：零安装门槛，依托微信庞大用户基础，大幅降低使用门槛。
4. **三角色分离设计**：用户/高校管理员/超级管理员职责清晰，权限矩阵严格控制，贴近真实业务场景。

### 6.2 不足与局限

| 不足点 | 说明 |
|-------|------|
| 区块链性能 | 以太坊主网 TPS 约 15-30，认证量大时可能出现拥堵 |
| Gas 成本 | 主网每次上链需真实 ETH，成本需控制 |
| 文件存储 | 当前设计中证书文件存于数据库/对象存储，未上链（有意为之，保护隐私） |
| 管理端 | 本次实现重点为用户端，管理端 UI 相对简单 |

### 6.3 未来展望

- **Layer2 / 联盟链**：采用 Arbitrum/Optimism 或 FISCO BCOS，降低 Gas 成本，提高 TPS。
- **批量上链**：将多条存证批量打包写入合约，降低单次成本。
- **去中心化存储**：证书文件存入 IPFS，链上存储文件 CID + hash，进一步去中心化。
- **跨链互认**：支持多条链上存证，满足国际认证互认需求。

### 6.4 面试高频问答提纲

**Q1：为什么要用区块链？不用行不行？**
> 不用也可以做（数据库+审计日志），但信任依赖中心化机构。区块链的价值在于：多方可验证、不可篡改、公开可查、无需信任第三方。本项目中链上存证使得任何持有原始字段的人都可独立验真，无需依赖系统平台。

**Q2：你上链了什么数据？hash 怎么算的？**
> 上链的是 canonical string 的 SHA-256 摘要（bytes32），不存明文。canonical string 格式为 `requestId|userId|schoolId|degree|major|graduationYear|timestamp`，Java 端用 `MessageDigest.getInstance("SHA-256")` 计算，与合约端的 keccak256 不同（选用 SHA-256 的原因是与标准 Java 库对齐，便于独立验真）。

**Q3：合约如何保证只有管理员能写入？**
> 合约使用 `onlyOwner` 修饰符，`owner` 在 `constructor` 中设为 `msg.sender`（即部署者）。只有 owner 地址发起的交易才能调用 `attest()`，否则 revert。`transferOwnership` 可以转移权限。

**Q4：数据库表为什么单独设 blockchain_records？**
> `Authentication_Requests` 是业务表，只存最终存证 hash；`Blockchain_Records` 是审计表，记录每次存证操作历史（包括 PENDING/SUCCESS/FAILED 状态），便于失败重试和完整审计追溯，避免业务表过度膨胀。

**Q5：上链失败怎么处理？**
> 后端采用异步补偿：写入 MySQL 时标记 `blockchain_status=PENDING`；定时任务轮询 PENDING 记录重试上链；成功后更新为 SUCCESS 并记录 txHash。

**Q6：如何防止学历信息被篡改后仍能通过验证？**
> 存证时的 hash 是原始字段的摘要。若任何字段被篡改，验证方重算 hash 后与链上存储的 hash 不匹配，自动检测到篡改。链上存储的 hash 本身不可修改（智能合约状态一旦上链不可逆）。

**Q7：你的创新点在哪里，和别人有什么不同？**
> 核心差异：本项目不仅做了系统设计，还**落地实现**了区块链存证部分（Solidity 合约 + Hardhat 本地开发环境 + Spring Boot web3j 集成），并采用"链上只存 hash，保护隐私"的设计原则，具备完整的开发→部署→API调用→验真的完整链路。
